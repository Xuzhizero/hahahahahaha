# 1、概述
## 1.1 研制背景与必要性
为支撑2026 年上半年通过 CCS 认证，并形成“先仿真、再海试”的研发闭环，必须建设一套可重复、可穷举、低成本的海上模拟仿真测试系统。当前存在的痛点包括：

1. 智能航行算法目前只是依赖理想环境。没有考虑风浪
2. 调试智能航行算法的人力和时间成本高，光是调节实船PID的参数就要一周的时间，日均经济成本接近2万元
3. 新的控制算法如果要验证极端操作情况（例如满舵后再反向满舵），在实船上不方便验证
4. 扩展性暂时没有考虑。 许多额外的装备，比如计程仪，这些架构没有搭载全面。如果要先通过购买实机测试效果和搭载必要性，然后再设计算法，开发周期会大大加长。仿真软件可以轻松添加具备相应功能的扩展模块，例如雷达、深度摄像头，参数均可自行调节
5. 避碰的来船设置、岛礁设置、浅滩设置，大雾大雨天气情况的设置，要么无法实现，要么设置比较麻烦。需要一个成熟的仿真软件很方便的提供这些场景
6. 未来涉及到更高级的基于学习的算法，就必须要先在仿真中先经受训练。

作为智能船舶研究院仿真实验室的重要组成部分，本项目对仿真实验室建设的战略意义如下：

1. 沉淀“模型—场景—数据—评测”统一规范与接口（ROS2/Simulink/UE5），成为实验室日常开发、联调、回归与培训的长期底座。
2. 将极端风浪、靠泊、动态避碰等高风险试验前置至室内，显著降低实船试验频次与安全风险，为 CCS 认证提供可追溯的仿真证据链与试验记录。
3. 提供标准场景库与评测流程，实现批量工况的自动化回归与可视化比对，把反复试错从海上转移到实验室。
4. 作为面向合作伙伴与主管部门的工程样机与展示窗口，支撑招引项目、联合攻关与人才培养。

同时，舟山海上测试场的方案论证已明确提出需配置虚实融合仿真能力，以支撑批量虚拟测试和对外技术服务。上述因素共同驱动本产品立项与研制落地。

## 1.2 技术定位
本产品定位为“面向自主航行的虚实融合仿真验证平台”，采用**ROS 2（算法决策）—Simulink（动力学与运动学高频推演）—UE5（高保真可视化与传感器仿真）**的端到端闭环架构：

+ **算法侧**：在两台 Ubuntu 主机上运行路径规划、避碰与控制等算法，统一 ROS 话题接口，便于与实船系统无缝迁移；
+ **模型侧**：Windows 主机以 Simulink 为时间主控，结合 MSS/ASV_Simulink 等成熟库，构建六自由度船体动力学与环境干扰模型，实现推力/力矩—姿态—状态量的高速递推；
+ **感知/可视化侧**：UE5 负责地形、风浪流、光照/天气、多目标通航等场景渲染，并通过“传感器映射”输出相机、激光雷达、IMU 等多源数据，必要时采样局部风浪信息回馈模型，实现宏观参数与微观粒子耦合的闭环仿真。  
平台强调**高保真、强扩展、易协同、低成本与快速迭代**，满足规控、感知定位、执行机构等多专业联调联试。 

## 1.3 市场定位
本产品面向“内外双向”市场：

+ **内部应用**：服务研究院及合作项目的算法开发/回归测试/极端工况验证，显著压缩实船调参时间与经费支出，形成算法—模型—数据一体化研制闭环；
+ **外部服务**：作为舟山海上测试场的重要能力模块，对无人船厂商、系统集成单位、海事/港航管理机构、海洋装备供应商及高校科研团队提供“虚实融合”的仿真测试与技术服务，支撑认证前验证、方案比选与场景复现；
+ **标准化产品化**：沉淀为可交付的软件/数据/场景包与服务包，支持按需定制传感器、海况、地形与通航规则，形成可复制的行业解决方案。 

## 1.4 产品名称与型号（暂定）
+ **产品名称（暂定）**：海上自主航行虚实融合仿真测试系统
+ **产品型号（暂定）**：**MSTS-100**（Maritime Simulation & Test System，首版/基础型）







# 2、国内外同类产品情况
## 2.1 类型划分与代表性产品
为便于对标，业内可大致分为三类：

**（1）培训级船舶/海事模拟器（STCW/IMO 合规）**  
典型产品包括 Kongsberg 的 K-Sim Navigation（提供高拟真水动力、环境与交互，可从全任务桥楼扩展到云端课程）与 Wärtsilä 的 NTPRO 5000（覆盖课堂、分任务到全任务桥楼配置，符合 STCW/IMO/ DNV 等主流认证，并已推出云仿真方案）。VSTEP 的 NAUTIS 系列面向课堂到便携/桌面化的多档配置。

**（2）工程分析级海洋/船舶动力学软件**  
用于海况—结构/船体相互作用建模、耐波性与系泊/缆索动力学等：DNV Sesam/HydroD（频域与时域耐波、波浪载荷分析）、ANSYS AQWA（多体绕射/辐射与 RAO 计算）、WAMIT（势流波浪相互作用老牌工具，提供多种授权方式）、MARIN aNySIM XMF（非线性时域多体耦合、DP/系泊/绞车等模块）、OrcaFlex、ProteusDS（面向系泊/渔业/海工装备动态分析）。

**（3）机器人/自主系统仿真平台（USV/多传感器/ROS 生态）**  
用于自主航行算法 V&V、HIL/SIL、合成数据：OSRF 的 VRX（Gazebo 海面/任务赛环境）、Unity 生态的 MARUS（开源、传感器/ROS2 适配、海面视觉高保真）。行业也在推进开放协同的数字孪生/共仿真标准（Open Simulation Platform，OSP）。

## 2.2 关键技术指标（选摘，对比维度）
+ **动力学与环境**：6DOF 船体动力学、风浪流耦合、浅水/岸壁效应、拖带/系泊/绞车/护舷等时域模型（aNySIM/OrcaFlex/ProteusDS 等）；培训级产品在操船视景、海区数据库与交互器材上更强。
+ **合规与认证**：培训级产品支持 IMO/STCW、DNV 类别认证，适配 DP、冰区、渔业等专项课程。
+ **开放集成**：自主级平台通常提供 ROS/ROS2 接口、可自定义传感器（IMU、GNSS、LiDAR、声呐、相机等），便于算法闭环与数据闭环。

## 2.3 市场格局与价格区间（2024–2025）
+ **市场规模与结构**（不同机构口径差异较大，仅作量级参考）：Mordor Intelligence 估计 2025 年海事模拟器市场约 **44.7 亿美元**、2030 年达 **61.7 亿美元**；属**半集中**市场，**Wärtsilä、KONGSBERG、VSTEP、Altair、BMT** 为主要厂商。Market.us 指出**船桥模拟器**为最大细分类型、**北美约 30%** 份额。
+ **典型价格**（受配置/投影系统/桥楼仿真硬件/座舱数量影响极大）：
    - **全任务桥楼级**：公开案例显示大型桥楼/投影系统成本可达 **数十万欧元量级**（如某大尺寸桥楼≈**€400,000** 报道）；美国 GSA 价目中“185° 平面屏桥楼”单项约 **$278,801**（不含全套系统）。
    - **云/桌面级**：Wärtsilä、Kongsberg 提供云端订阅；NAUTIS Home 等桌面产品面向个人/基础训练，订阅价显著低于全任务方案（示例报道为**每月欧元级**）。
    - **工程分析软件授权**：多采用节点锁/浮动/租赁等模式（如 WAMIT 提供月/年租授权；OrcaFlex 提供本地/浮动许可）。

## 2.4 发展趋势与应用前景
+ **数字孪生与共仿真**：以 OSP 为代表，推动设备/系统模型复用、跨企业协同验证，贯穿设计—建造—运维。
+ **云化与混合教学**：云仿真获得 DNV Class D 等认证后，线上线下混合训练成为常态，支撑大规模与分布式教学。
+ **自主与智能化测试**：ROS2 生态（VRX/MARUS）持续成熟，支持多传感器、域随机化与合成数据，降低海试成本与风险，缩短算法迭代周期。
+ **专项/场景化能力**：诸如拖带、冰区、渔业、溢油应急、海警快速艇训练等专项场景需求增长（新装置与新能源船型也提升训练复杂度）。

## 2.5 应用场景与系统内关系
+ **训练/资质**：海员基础到高级值班、BRM/BTM、DP、冰区/渔业/拖轮等专题训练与考核；云平台支持远程实训与复训。
+ **工程/运维**：新港区/新航道评估、系泊/靠离泊方案优化、风浪载荷评估、设备虚拟调试与虚拟试航。
+ **自主系统**：算法开发—仿真—硬件在环—实船验证的闭环，打通感知（合成数据）、定位/导航、路径规划与航行规则遵循。  
在系统架构中，仿真平台位于**控制算法与虚实环境之间的中间层**：向上连接任务/控制（ROS2/自研控制器/仿真中间件），向下对接**高保真海况/船体/系泊动力学**与**传感器合成**，并通过**云端资源与工况库**支撑批量回放/回归测试；在工程场景中可与 HydroD/AQWA/WAMIT 等离线求解器联动生成载荷/RAO，再用于时域验证（共仿真）。

## 2.6 拟研发产品的对比优势与必要性
**对比优势**（与上述三类产品对标）：

1. **“训练可用 + 自主可研 + 工程可证”的三用融合**：在维持培训级易用/云化与丰富场景的同时，提供 ROS2/HIL 接口与工程级动力学接口（与 AQWA/WAMIT/SESAM/HydroD 结果对接或共仿，支撑设计—验证闭环）。现有培训级产品在开放接口与算法闭环方面受限，而工程级工具在实时性/交互和教学合规上不足。
2. **成本与交付效率**：采用模块化软硬件与云端弹性算力，桌面—集群—全任务可伸缩，对比传统全任务桥楼一次性投入（公开价目显示单项桥楼构型即达数十万美元）可显著降低单位训练席位/测试算例成本。
3. **标准化与生态兼容**：面向 OSP/FMI/ROS2 的模型与接口标准，便于与船厂、设备商、院校和主管机关共享模型与场景，缩短验标周期。
4. **高保真合成数据与安全性**：以 Unity/GPU 加速与真实海况库生成感知训练数据，结合场景风控规则与航行法规库，提高自主系统“上船前”的可验证性并降低海试风险。

**研发必要性**：

+ **政策/培训刚需**：国内培训大纲对复杂海况/受限水域/VTS 等训练有明确要求，培养规模扩大与新型船舶（新能源、智能航行）推动模拟训练能力升级。
+ **自主系统迭代需求**：海试周期长、成本高、风险大，需在仿真端完成大批量情景回归与危险工况验证（行业普遍向云仿真与数字孪生迁移）。
+ **产业协同**：开放共仿真与标准化模型有助于船厂/设备商/高校形成联合创新与验证闭环，缩短交付周期并促进国产软硬件生态发展。



# 3、技术性能要求
## 3.1 总体与实时性指标 
1. **仿真步长与实时因子**
+ 指标：动力学主时钟**Ts=20 ms（50 Hz）**；整链实时因子 **RTF≥1.0**（1×实时）持续 2 h。
+ 验收：在标准硬件基线下（Win11+i9-13900K/64 GB/RTX 4080；Ubuntu 22.04+i7-12700/32 GB×2），记录 2 h 内每 10 s 的 RTF，**P99≥1.0**。
+ 依据：系统以 **Simulink 为时间主控**，与 ROS 2、UE5 保持统一步长与同步。
1. **端到端链路时延（ROS 2 → Simulink → UE5 → ROS 2）**
+ 指标：**中位数 ≤20 ms，P95 ≤35 ms，P99 ≤50 ms**；抖动（P95-P50）≤15 ms。
+ 验收：在同一子网，回环时戳比对链路报文。
1. **可视化帧率与一致性**
+ 指标：UE5 视景 **FPS≥60（P95）**；**无帧-步长漂移**（每 10 min 采样，步长累计误差≤1 帧）。
+ 验收：UE5 帧时间与 Simulink 步长同步日志对比。
1. **长稳性**
+ 指标：**8 h 连续运行 0 崩溃**；进程常驻内存增长 **≤5%/h**。
+ 验收：系统监控+日志审计。

## 3.2 动力学与环境建模 
1. **船体与执行机构**
+ 指标：**6 DoF** 非线性动力学；至少支持 **舵-桨/差速双桨/双直翼舵桨**三类执行机构；极限操纵（**满舵→反满舵**）可重现实验轨迹。
+ 验收：给定三种推进布置参数库，复现实验型谱，偏差≤规定阈值。 
1. **风浪流环境**
+ 指标：宏观参数范围：风速 0–25 m/s；**Hs 0–6 m**；主频 **0.05–0.5 Hz**；流速 0–2 m/s；方向 0–360°；支持 **Pierson-Moskowitz/JONSWAP** 波谱；**局部粒子采样回传**用于干扰力叠加。
+ 验收：参数扫描+谱一致性检验；局部采样与宏观参数关联度 R²≥0.9。 
1. **极端工况与场景**
+ 指标：支持**随机/预设来船**、**浮标/岛礁/电子围栏（含海底坡度）**、**狭窄水域/港区靠泊**等；最少 **≥30** 个可复现实例。
+ 验收：任务库运行通过率 100%。

## 3.3 传感器合成与数据
1. **相机**：RGB/深度/语义分割；**分辨率 1920×1080@30 FPS（P95）**；运动模糊、HDR、天气响应可开关。
2. **LiDAR**：32/64 线；**≥200 k pts/frame@10 Hz**；支持雨雾衰减/噪声模型参数化。
3. **IMU/GNSS**：IMU **200 Hz**；GNSS **10 Hz**；可注入偏置/噪声。
+ 验收：对比 UE5 渲染/雾效参数与传感器回放一致性。 

## 3.4 协同与接口 
1. **ROS 2 原生**：跨三机部署（Ubuntu×2 + Win×1），通过 ROS Topic/Clock 同步；支持 **< 1 Gbps** 总链路带宽下稳定运行。
+ 验收：跨机话题延迟/丢包率 **≤0.1%**。
1. **模型互联**：可从外部水动力求解器导入 **RAO/波浪力**并参与时域仿真（接口预留）。
+ 验收：导入样例并回放。

## 3.5 可视化与地形 
1. **地形/海域**：支持高度图 RAW/PNG、FBX/OBJ、Datasmith 导入；至少 **≥5** 个实海域（含舟山）预置关卡。
+ 验收：导入流程全自动化通过。

## 3.6 记录、复现与评测 
1. **全链路录制**：状态/指令/传感器/事件**统一时戳**；磁盘写入吞吐 **≥200 MB/s**；掉样 **≤0.1%**。
2. **复现实验**：同一随机种子与输入下，轨迹 Hausdorff 距离 **≤0.5 m**、姿态误差 **≤1°**。

## 3.9 本产品相对对标项的技术优势

# 4、初步技术方案
## 4.1 方案总览（架构与部署）
![仿真流程图](https://cdn.nlark.com/yuque/0/2025/png/2408029/1753324938961-9206d663-6b88-4c7c-856f-c21c8ec29a8c.png?x-oss-process=image%2Fformat%2Cwebp)

本系统采用“**ROS 2（算法）—Simulink（动力学/时间主控）—UE5（可视化/传感器仿真）**”的端到端闭环架构：两台 Ubuntu 主机分别承载**制导/导航/控制**与**感知定位**算法；一台 Windows 主机承载 **Simulink** 动力学与 **UE5** 场景/传感器仿真。  
链路为 **ROS→Simulink→UE5→ROS**：ROS 下发推力/力矩，Simulink 推演 6DoF 状态并同步至 UE5，同时接收 UE5 的**局部风浪粒子采样**回传，最终再将状态/传感器数据回灌 ROS。  
全链统一由 **Simulink 作为时间主控/发布 /clock**，保证跨机同步；控制/感知/路径算法均以 **ROS 2** 实现，便于直接迁移至实船。

## 4.2 数据流与模块关系
+ **环境渲染/可视化分系统**：接收船体状态，渲染海面/障碍/天气，输出场景画面与传感器视图及环境参数。
+ **模型递推分系统**：保持时间主控；接收外部扰动与推进/舵指令，输出船体状态至可视化与算法分系统；同时负责**地形/风浪宏观参数**下发 UE5。 
+ **智能航行算法分系统**：执行避障、路径规划、运动控制、感知融合等，订阅船体状态并发布执行指令。

## 4.3 分系统初步方案
### 4.3.1 模型递推分系统（Simulink）
+ **执行机构与干扰耦合**：接收 ROS 指令，换算推进力/力矩；依据 **UE5 局部采样**与宏观风浪参数计算环境干扰力矩。 
+ **6DoF 非线性动力学与运动学**：将推进/干扰耦合入 6DoF 模型，积分得到位姿并回传。
+ **参考实现**（已在文档中选型）：**ASV_Simulink + MSS** 体系，含 Thrust / Waves&Crossflow / Hydrodynamics / Hydrostatic 等子模块与 **6DOF(in CO)** 输出链路。

### 4.3.2 可视化与传感器仿真分系统（UE5）
+ **地形/海况/来船**：依据宏观风浪参数生成大尺度海况，并在船体周边布置采样点获取**局部风浪信息**回传；可脚本化驱动**来船**目标。 
+ **光照/天气**：Sky Atmosphere、Volumetric Clouds、Exponential Height Fog 等组件参数化控制，支持雨雾等低能见度工况。 
+ **传感器级仿真**：在完整场景中生成 RGB/深度/语义、LiDAR 点云、IMU 原始测量等并回灌 Simulink/ROS。
+ **船体映射（已实现）**：保证物理位姿与渲染严格一致，支撑算法可视化调试。

### 4.3.3 智能航行算法分系统（ROS 2）
+ **部署与接口**：算法运行于两台 Ubuntu 主机；通过 ROS Topic 与 Simulink/UE5 通信，职责涵盖**循迹/避碰/靠泊**与**感知融合**。 

## 4.4 关键技术路径与实现要点
1. **统一时钟与实时同步**：以 Simulink 为全链路**时间主控**发布 /clock；算法、模型与可视化按照统一步长与节拍执行。
2. **风浪场景的宏观—微观闭环**：Simulink 下发**宏观风速/显著波高/波谱**参数，UE5 生成海况并在船体周边**局部采样**回传，叠加入动力学干扰。
3. **传感器/船体映射**：利用 UE5 传感器与姿态接口，将 Simulink 的位姿/参数映射至 UE5 并获得传感器数据回灌。 
4. **多机 ROS 通信（已实现）**：统一 ROS_DOMAIN_ID 与 RMW，通过 Simulink ROS2 Publish/Subscribe 完成跨平台节点互通与话题对接。

## 4.5 可行性分析
### 4.5.1 技术可行性
+ **组件成熟度**：ASV_Simulink/MSS 提供开箱船体/环境动力学与接口模板；UE5 提供成熟的海况、天气与传感器渲染链路；ROS 2 提供稳定的分布式通信体系。上述在文档中**已选型并串接**。 
+ **关键环节已打通**：多机通信、船体映射已实现并用于闭环调试，验证了跨机同步与位姿一致性。 

### 4.5.2 需求满足度
系统定性指标侧重**全面性、数据易得、传感器高保真、易扩展、可视化保真、实时性、快速迭代、协同与成本**；本方案在架构与实现上逐项对齐（统一时钟/多机协同、宏微耦合海况、参数化传感器与地形导入、ROS 原生接口），可覆盖“规划控制/感知定位/靠泊/避碰”等核心验证场景。

### 4.5.3 实施与运维可行性
+ **部署**：三机部署与局域网 ROS 组网，按文档流程即可复现；地形/海况/光照可参数化导入与批量切换，降低运维复杂度。 
+ **扩展**：新增传感器/执行机构通过标准接口与蓝图脚本快速集成；外部动力学结果（如 RAO/波浪力）可通过现有接口注入（文档已留出数据通道与流程）。
+ **协同**：全链数据与时序统一，便于算法、模型与可视化团队并行联调（投屏/回放见数据流与输出）。

### 4.5.4 风险与对策
+ **实时一致性风险**：跨机网络抖动导致 RTF 波动 → 采用固定步长、对关键话题启用 QoS（可靠/保持）、在 UE5 侧启用帧—步长对齐校验。
+ **模型与视觉失配**：海况材质参数与动力学谱参数不一致 → 用一组**宏观参数**驱动两端，并对局部采样做统计一致性校验。
+ **场景/数据规模过大**：高分地形与多传感器并发 → 分层加载地形与传感器 ROI 采样，必要时采用离线预渲染/分布式录制。

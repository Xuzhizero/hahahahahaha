# <font style="color:rgb(28, 31, 35);">提问</font>
<font style="color:rgb(28, 31, 35);">那显示器有帧率一说吗？如果有，那如果一个视频是30fps的，显示器是60fps的，最终的帧率是多少呢</font>

<font style="color:rgb(28, 31, 35);"></font>

# <font style="color:rgb(28, 31, 35);">回答</font>
显示器有帧率的概念，且显示器帧率与视频帧率是两个不同维度的参数，最终呈现的画面帧率需结合两者的关系来判断：



### 一、显示器的“帧率”是什么？
显示器的帧率（通常称为**刷新率**，单位Hz）指的是显示器**每秒能刷新画面的最大次数**，是显示器的硬件性能指标。  
例如：  

+ 60Hz显示器：每秒最多能刷新60次画面；  
+ 144Hz显示器：每秒最多能刷新144次画面。  
刷新率越高，显示器显示动态画面时的“流畅感上限”越高（尤其对游戏、快速运动画面更明显）。



### 二、视频帧率与显示器刷新率的关系
当视频帧率（如30fps）与显示器刷新率（如60Hz）不一致时，最终呈现的画面帧率由**视频本身的帧率决定**，但显示器会通过特定方式“适配”画面输出，具体分两种情况：



#### 1. 最终呈现的“有效帧率”= 视频帧率（30fps）
显示器会将视频的30帧画面**重复显示**以匹配自身刷新率：  

+ 60Hz显示器播放30fps视频时，每帧画面会被显示2次（30帧×2=60次刷新），确保每秒刷新次数与显示器刷新率一致；  
+ 此时画面的“内容更新频率”仍是30fps（每秒只有30个新画面），但显示器通过重复帧避免了画面撕裂，观感上依然是30fps的流畅度（不会因显示器刷新率高而变“更流畅”）。



#### 2. 特殊技术：动态插帧（提升观感，但非真实帧率）
部分高端显示器或电视支持**MEMC（运动补偿技术）**，原理是通过算法在视频原有帧之间**插入“虚拟帧”**：  

+ 例如30fps视频在60Hz显示器上，MEMC会计算相邻两帧的运动轨迹，插入30个虚拟帧，使总画面达到60fps；  
+ 此时画面的“显示帧率”是60fps，但“原始内容帧率”仍是30fps（虚拟帧不增加新信息）。这种技术能让运动画面更顺滑，但本质是对原有画面的优化，而非改变视频本身的帧率。



### 三、总结
+ 显示器有帧率（刷新率），但它是**硬件的“输出能力上限”**，不改变视频本身的内容帧率；  
+ 30fps视频在60Hz显示器上播放时，**最终的“内容帧率”仍是30fps**，显示器通过重复帧或插帧适配输出，观感流畅度取决于视频本身的30fps，而非显示器的60Hz；  
+ 只有当视频帧率与显示器刷新率匹配（如60fps视频在60Hz显示器上播放），且无重复帧时，才能完全发挥显示器的刷新率优势，画面会更流畅。

# 继续提问
<font style="color:rgb(28, 31, 35);">如果一个视频的帧率是60fps，这个视频通过一个30fps的投屏设备投屏到另一个刷新率足够高的显示器上（比如120HZ）， 那么显示器的最终帧率是多少？</font>

<font style="color:rgb(28, 31, 35);"></font>

# <font style="color:rgb(28, 31, 35);">回答</font>
当60fps的视频通过30fps的投屏设备投屏到120Hz显示器时，**最终显示器呈现的画面帧率会被限制为30fps**，具体原因如下：



### 一、投屏设备是“帧率瓶颈”
投屏设备的“30fps”指的是它**每秒最多能传输30帧画面**（硬件性能或传输协议的限制）。  
此时，即使原始视频是60fps（每秒60个画面），投屏设备也会**主动丢弃一半的帧**（仅保留30帧/秒），只向显示器传输30fps的画面数据。  



### 二、显示器的高刷新率无法“弥补”上游帧率限制
120Hz显示器的作用是“每秒最多能刷新120次画面”，但它只能基于**接收到的画面数据**进行刷新：  

+ 当接收到的是30fps的画面时，显示器会将每帧画面**重复显示4次**（30帧×4=120次刷新），以匹配自身120Hz的刷新率；  
+ 此时画面的“内容更新频率”仍是30fps（每秒只有30个新画面），显示器的高刷新率无法凭空增加原始画面的帧数，因此最终呈现的**有效帧率仍是30fps**。



### 总结
整个链路的帧率由**性能最弱的环节（投屏设备）决定**：60fps视频→30fps投屏设备（帧率被压缩）→120Hz显示器（被动适配30fps画面），最终显示器呈现的帧率为30fps。



# 继续提问
<font style="color:rgb(28, 31, 35);">“投屏设备也会主动丢弃一半的帧” 是丢弃哪一半？ 另外，如果投屏帧率偏低，是否会导致投屏终端看到的内容越来越滞后于画面来源端？</font>

# <font style="color:rgb(28, 31, 35);">回答</font>
关于投屏设备丢帧的规则和帧率偏低导致的滞后问题，具体可以这样理解：



### 一、投屏设备“丢弃哪一半帧”？
投屏设备（如30fps）处理60fps视频时，丢帧的核心逻辑是**“均匀丢弃”以保证画面时间线的连续性**，而非随机丢弃某部分帧。具体方式有两种：  

1. **间隔丢弃**：每两帧中丢弃一帧，例如保留第1、3、5……59帧，丢弃第2、4、6……60帧，确保每秒保留30帧，且相邻帧的时间间隔仍为1/30秒（与原视频的时间节奏一致）。  
2. **按时间切片保留**：将1秒的60帧画面分成30个“时间窗口”（每个窗口约33毫秒），每个窗口内只保留1帧（通常是中间帧或首帧），避免某段时间画面完全丢失。

这两种方式的目的都是**保证画面内容的时间顺序不混乱**，只是减少了细节更新频率，不会出现“跳帧”或“画面顺序颠倒”的情况。



### 二、投屏帧率偏低是否会导致“滞后越来越严重”？
**不会出现“越来越滞后”（累积延迟），但会有“固定滞后”**，原因如下：  

1. **帧率与延迟的关系**：  
帧率偏低（如30fps）影响的是画面的“流畅度”（每秒更新次数少），但与“滞后量”（画面来源端与投屏终端的时间差）无直接关联。  
滞后主要来自**数据传输、编码解码的耗时**（例如投屏时需要压缩视频、通过网络传输、终端解压渲染，这个过程会产生几十到几百毫秒的固定延迟），但这个延迟是“一次性”的，不会随时间累积。  
2. **为什么不会累积滞后？**  
投屏设备会严格按照“实时同步”原则处理画面：即使只传输30fps，每帧画面也会对应来源端的某个时间点，终端会在接收到帧后立即显示，不会“囤积”帧再批量输出。因此，滞后量会稳定在一个固定值（如100ms），而非随播放时间增加（比如不会从100ms变成1秒）。  
3. **特殊情况：极端低帧率可能加剧“卡顿感”**  
若帧率过低（如10fps以下），画面更新太慢，会让人主观感觉“反应迟钝”（例如操作来源端的鼠标，投屏终端要过很久才显示移动），但这是“流畅度差”导致的错觉，本质仍是固定延迟，而非累积滞后。



### 总结
+ 投屏设备丢帧是**均匀丢弃以保持时间连续性**，避免画面混乱；  
+ 帧率偏低会影响流畅度，但**不会导致滞后越来越严重**，滞后主要来自传输和编码的固定耗时，且是稳定的。

# 继续提问
<font style="color:rgb(28, 31, 35);">可是投屏设备怎么知道视频画面是多少帧呢？它有办法测出来，然后再根据测出来的帧率灵活调整丢弃的帧的“策略”吗？</font>

# <font style="color:rgb(28, 31, 35);">回答</font>
投屏设备主要通过解码视频数据和分析画面特征来获取视频帧率，并能够根据帧率等因素灵活调整丢帧策略，具体如下：

+ **通过解码视频数据获取帧率**：视频文件本身会携带帧率等元数据信息，投屏设备在接收到视频数据后，会对其进行解码。在解码过程中，设备可以从视频流的头部信息或相关参数字段中读取到原始视频的帧率，例如常见的25fps、30fps或60fps等。
+ **分析画面特征确定帧率**：部分投屏设备还会通过分析画面特征来辅助确定帧率或调整投屏帧率。具体方法是采集待投屏的图像帧序列，计算每相邻两帧图像间的亮度差，得到亮度差序列。若连续多个亮度差大于某一阈值，可判断为动态画面，设置较高的投屏帧率；若亮度差均小于或等于阈值，可认为是静态画面，设置较低的投屏帧率。
+ **根据帧率调整丢帧策略**：投屏设备确定视频帧率后，会基于此确定投屏间隔时间。然后，设备获取图像帧序列中最新采集的目标图像帧，计算其与上一次发送的图像帧间的实际间隔时间。若实际间隔时间小于投屏间隔时间，就丢弃目标图像帧；若大于或等于，则将其发送至接收终端。此外，投屏设备还会考虑传输带宽等因素，若带宽不足，可能会直接以固定帧率发送图像帧，以保证投屏的稳定性。
